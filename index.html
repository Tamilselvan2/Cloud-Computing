Stored Program Computer Architecture

The Von Neumann model — instructions and data share the same memory. The CPU fetches, decodes, and executes instructions sequentially. This is the backbone of all modern computing, extended in HPC with parallelism.

Components

▸Input Unit — accepts data & instructions
▸Memory Unit — stores program & data
▸CPU (Control Unit + ALU)
▸Output Unit — produces results

Fetch-Decode-Execute Cycle

Fetch → Decode → Execute → Store

The CPU repeats this cycle for every instruction. In HPC, this is pipelined and parallelized.

Von Neumann Bottleneck

Same bus is used for data and instructions, causing limited transfer speed. HPC overcomes this with separate instruction/data caches, high-speed interconnects, and distributed memory systems.

10-Mark Important

Memory Hierarchy

CPU is very fast but memory is slower. A hierarchy balances speed, capacity, and cost. As we move down: speed decreases, capacity increases, cost per bit decreases.

Temporal Locality

Recently used data is likely to be reused soon. Cache stores this data to avoid re-fetching from slower memory.

Spatial Locality

Data stored physically close together (like arrays) is likely to be accessed together. Cache lines prefetch neighboring data.

Very Important — 10 Mark

Flynn's Taxonomy

Michael J. Flynn (1966) classified parallel architectures based on instruction and data streams.

1

SISD

Single Instruction, Single Data

One processor executes one instruction on one data item. Sequential execution.

Example: Traditional single-core PC
2

SIMD

Single Instruction, Multiple Data

One instruction controls multiple processing elements. Same operation on different data simultaneously.

Example: GPUs, Vector processors
3

MISD

Multiple Instruction, Single Data

Multiple instructions operate on the same data stream. Rarely used in practice.

Example: Fault-tolerant space systems
4

MIMD

Multiple Instruction, Multiple Data

Multiple processors execute different instructions on different data. Most common in HPC.

Example: Multicore CPUs, Clusters

Multiple distinct CPU cores on a single chip. Each core executes instructions independently, sharing L3 cache and main memory.

Example: Quad-core = 4 cores

Single core executes multiple threads. If Thread A stalls waiting for memory, the CPU instantly switches to Thread B.

Example: Intel Hyper-Threading

Use: Scientific computing, GPUs
Very Important

Amdahl's Law

Fixed workload. Speedup is strictly limited by the sequential (non-parallelizable) fraction of the code.

Speedup Formula
S(N) = 1 / [(1 − P) + P/N]
•S(N) = Theoretical maximum speedup
•P = Parallel portion fraction
•N = Number of processors

If 5% is serial, max speedup is 20× — no matter how many processors you add.

Very Important

Gustafson's Law

Scaled workload. Execution time is fixed; problem size grows with processors. The sequential part shrinks proportionally.

Speedup Formula
S(N) = N − α(N − 1)
•α = Serial fraction
•N = Number of processors

Optimistic view: more processors → bigger problems → efficiency stays high.

Amdahl vs Gustafson

OutlookPessimisticOptimistic
Key InsightSerial part is the bottleneckParallel part grows with problem

OpenMP — Shared Memory Parallel Programming

OpenMP (Open Multi-Processing) is an API for shared-memory multiprocessing in C/C++. It uses compiler directives (#pragma omp) and follows the Fork-Join model.

Fork-Join Model

1
Master thread starts execution
2
Fork: multiple worker threads created
3
Threads execute parallel region
4
Join: threads synchronize and merge

Data Scoping

✓
shared

All threads read/write same memory

✓
private

Each thread has its own copy

✓
firstprivate

Initialized private copy

Basic OpenMP Program

#include <stdio.h> #include <omp.h> int main() { #pragma omp parallel { printf("Hello World\n"); } return 0; }

Compilation: gcc file.c -fopenmp

Key Directives

#pragma omp parallel for

Divides loop iterations among threads automatically

#pragma omp critical

Only one thread executes this block at a time

#pragma omp barrier

All threads wait until everyone reaches this point

#pragma omp parallel for reduction(+:sum)

Safely combines local results into global result

Loop Scheduling

Static

Chunks assigned to threads before loop starts. Predictable.

Dynamic

Threads grab chunks on the fly as they finish. Best for unequal work.

Guided

Chunk size decreases over time. Balanced approach.

#pragma omp parallel for schedule(dynamic)
Case Study — 10 Mark

Parallel Jacobi Algorithm (OpenMP)

Used to solve systems of linear equations iteratively. Highly parallelizable because calculating the current step doesn't depend on other calculations in the same step.

Jacobi Iteration Formula
xi(k+1) = (1/aii) × [ bi − Σj≠i aij × xj(k) ]

OpenMP Implementation

#pragma omp parallel for for(int i = 0; i < n; i++) { double sigma = 0; for(int j = 0; j < n; j++) { if(i != j) sigma += A[i][j] * x[j]; } xnew[i] = (b[i] - sigma) / A[i][i]; }

Key: Read from "old" grid, write to "new" grid to avoid race conditions.

Performance Pitfalls

OpenMP Performance Pitfalls

False Sharing

Two threads update different variables that accidentally reside on the same cache line. CPU constantly invalidates and reloads cache — massive performance killer.

Load Imbalance

Some processors finish tasks earlier while others are still working. Leads to idle CPU time and reduced overall efficiency.

Synchronization Overhead

Frequent use of locks, barriers, and critical sections causes threads to wait for each other unnecessarily.

Communication Overhead

Excessive data transfer between processors. High latency and bandwidth limitations reduce speed.

MPI — Distributed Memory Programming

Message Passing Interface (MPI) is the standard for distributed systems. Processes have separate memories and communicate by explicitly sending messages over a network.

Basic MPI Program

#include <mpi.h> #include <stdio.h> int main(int argc, char *argv[]) { MPI_Init(&argc, &argv); int rank, size; MPI_Comm_rank(MPI_COMM_WORLD, &rank); MPI_Comm_size(MPI_COMM_WORLD, &size); printf("Hello from process %d\n", rank); MPI_Finalize(); return 0; }

Compilation: mpicc file.c   Run: mpirun -np 4 ./a.out

Point-to-Point Communication

Direct message between exactly two processes.

MPI_Send(&data, 1, MPI_INT, dest, tag, comm); MPI_Recv(&data, 1, MPI_INT, src, tag, comm, &status);

Non-Blocking Communication

Program doesn't wait for transfer to finish. Overlaps communication with computation.

MPI_Isend(&data, ...); // Immediate send MPI_Irecv(&data, ...); // Immediate receive

Collective Communication

Case Study — 10 Mark

MPI Parallelization of Jacobi Solver

The global grid is sliced into horizontal chunks. Each process gets a chunk. To calculate boundary edges, processes need data from neighbors.

The Solution: Halo / Ghost Cells

Each processor allocates an extra empty row around its data called a "Ghost Cell." Before computing the next iteration, processors use Point-to-Point messages to swap their boundary rows with neighbors, filling the ghost cells.

Virtual Topologies

MPI arranges processes into logical grids (e.g., Cartesian) so processors know their North, South, East, West neighbors for easy halo exchange.

MPI_Cart_create(...)

Performance Properties

MPI performance depends heavily on network bandwidth and message latency. Minimizing communication volume is key to scaling.

Very Important Program

Matrix Addition using CUDA

#include <stdio.h> __global__ void add(int *a, int *b, int *c) { int id = threadIdx.x; c[id] = a[id] + b[id]; } int main() { int a[5] = {1,2,3,4,5}; int b[5] = {5,4,3,2,1}; int c[5]; int *d_a, *d_b, *d_c; cudaMalloc(&d_a, 5*sizeof(int)); cudaMalloc(&d_b, 5*sizeof(int)); cudaMalloc(&d_c, 5*sizeof(int)); cudaMemcpy(d_a, a, 5*sizeof(int), cudaMemcpyHostToDevice); cudaMemcpy(d_b, b, 5*sizeof(int), cudaMemcpyHostToDevice); add<<<1, 5>>>(d_a, d_b, d_c); cudaMemcpy(c, d_c, 5*sizeof(int), cudaMemcpyDeviceToHost); for(int i=0; i<5; i++) printf("%d ", c[i]); cudaFree(d_a); cudaFree(d_b); cudaFree(d_c); return 0; }

Performance Considerations

Memory Coalescing

Consecutive threads accessing consecutive addresses = one fast transaction. Random access cripples performance.

Branch Divergence

Threads in a Warp (group of 32) taking different if/else paths forces sequential execution. Kill performance.

Numerical Issues

Floating-point math isn't perfectly associative. Parallel sums may have tiny rounding differences vs CPU.

Advanced

Sparse Matrix Computation

Matrices mostly filled with zeros. Storing all zeros wastes GPU memory. Compressed using CSR (Compressed Sparse Row) format.

•Store only non-zero values and their coordinates
•Regularization: organize data so memory access is coalesced
•Equal load balancing among threads
Advanced

CUDA Dynamic Parallelism

Normally, CPU launches GPU kernels. Dynamic Parallelism allows a GPU thread to launch another GPU grid/kernel directly — without talking to the CPU.

Perfect for recursive algorithms (Quad-Trees, decision trees) where the GPU discovers it needs more work as it goes.

Application Case Studies

Real-World CUDA Applications

🧬

Molecular Visualization

N-body simulations — every atom exerts force on every other atom. GPUs assign one thread per atom to calculate total forces, enabling real-time chemical visualization.

🤖

Machine Learning

Deep learning = massive Matrix Multiplication (GEMM). Every output cell is independent, so GPUs calculate them all simultaneously. This is why AI exploded with GPUs.
